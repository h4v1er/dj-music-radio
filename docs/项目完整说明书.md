# DJ 音乐电台项目完整说明书

本文档用于课程项目说明、答辩 PPT、分工复盘和后续二次整合。内容按当前 `dev-chat` 分支真实代码整理，不按最初设想夸大已完成范围。

## 1. 项目定位

DJ 音乐电台是一个桌面式网页音乐控制台。用户在一个页面里完成：

- 和 AI DJ 聊天；
- 查询天气、时间、当前位置；
- 根据心情、天气、场景或偏好获取歌曲推荐；
- 播放本地歌库或网易云搜索结果；
- 刷新页面后按用户恢复播放队列和播放模式；
- 导入网易云歌单；
- 查看歌词、歌曲情绪画像、收藏、播放历史；
- 查看热门榜、今日推荐、偏好标签。

项目不是传统多页面后台管理系统，而是一个“音乐助手工具”：左侧聊天、中间播放器、右侧推荐、顶部时间天气、底部用户状态。

```text
  |-- WeatherService: 和风天气

  |-- MySQL mall.chat_history

module-music
  |-- MySQL dj_music_radio: song/playlist/favorite/history/queue/player_state/emotion/taste
  |-- RabbitMQ: music.playlist.import
  |-- NeteaseCloudMusicApi 3000
  |-- DeepSeek: 歌词情绪分析

  |-- Redis ZSET: song:hot:rank
  |-- MySQL mall: user_behavior/daily_recommend

  |-- MySQL mall.dj_user
  |-- BCrypt 密码哈希
  |-- JWT 签发和校验
  |-- OpenFeign -> module-music 收藏/历史
```

## 5. module-chat：智能对话 DJ

### 5.1 用户可见能力

- 普通自然聊天；
- 根据明确音乐需求推荐歌曲；
- 根据天气、时间、当前位置参与推荐；
- 查询任意城市当前天气；
- 未指定城市时可通过浏览器定位工具获取当前位置；
- AI 推荐出的歌曲在聊天气泡中显示为可点击条目，点击后由播放器播放。

### 5.2 后端核心流程

```text
用户消息
  -> ChatService 读取最近历史和前端 context.location
  -> DeepSeekChatClient.planToolUse 规划工具
  -> 如需 location.current 且无定位上下文，WebSocket 返回 tool_request
  -> 前端执行浏览器定位，再带 context.location 重发
  -> 后端执行工具：music / rec / weather / time / location
  -> DeepSeekChatClient.composeReply 根据工具结果和候选歌曲生成回复
  -> ChatService 保存历史并返回 reply、selectedSongs、toolCalls
```

### 5.3 工具白名单

| 工具 | 执行端 | 说明 |
|:--|:--|:--|
| `music.search` | 后端 | 调 `module-music /music/song/search` |
| `music.catalog` | 后端 | 调 `module-music /music/song/list` 获取候选池 |
| `music.neteaseSearch` | 后端 | 调 `module-music /music/netease/search` |
| `rec.daily` | 后端 | 调 `module-rec /rec/daily` |
| `rec.hot` | 后端 | 调 `module-rec /rec/hot` |
| `rec.preferences` | 后端 | 调 `module-rec /rec/preferences` |
| `weather.now` | 后端 | 调和风天气接口 |
| `time.current` | 后端 | 返回当前日期时间星期时区 |
| `location.current` | 前端 | 浏览器定位，需要用户授权 |

当前没有通用 `web.search` 工具，AI 不能随意联网抓取。

### 5.4 兜底逻辑

- `DEEPSEEK_API_KEY` 未配置：使用本地启发式工具规划；
- DeepSeek 调用失败：返回空 AI 回复，ChatService 用本地兜底话术；
- 数据库不可用：对话历史写内存，30 秒后重试数据库；
- music/rec 工具无结果：尝试补充本地歌曲候选池；
- 天气 API 不可用：返回 `source=demo` 和明确 `message`。

## 6. module-music：音乐中心

### 6.1 后端能力

- 歌曲分页、详情、搜索、流派列表；
- 播放详情时增加播放次数；
- 保存歌词；
- 手动/批量情绪分析；
- 歌单创建、更新、删除、歌曲添加/移除、排序；
- 歌单导入，通过 RabbitMQ 异步消费；
- 网易云搜索、播放 URL、详情、歌词、歌单详情、封面代理；
- 收藏、取消收藏、收藏检查、收藏列表；
- 播放历史记录和查询；
- 播放队列、当前歌曲、播放模式按用户持久化；
- 用户品味画像和情绪分布。

### 6.2 数据来源

`song` 表目前不再预置固定测试歌曲，主要通过网易云歌单导入或手动 JSON 导入形成数据。网易云结果的播放能力取决于 `NeteaseCloudMusicApi` 能否返回 URL；VIP、版权限制、无版权歌曲返回空 URL 是正常外部限制。

### 6.3 情绪分析

分析路径：

```text
歌词 LRC
  -> 优先 DeepSeek 歌词情绪分析
  -> 失败后用 LyricsAnalyzer 关键词规则分析
  -> 写入 song_emotion
  -> 回写 song.emotion_tags / song.emotion_analyzed
  -> UserTasteService 根据收藏、播放、导入更新用户品味
```

情绪体系包含 20 个中文标签，按柔和抒情、深沉内省、高能释放、复杂暗黑、叙事人文五类组织。

### 6.4 外部依赖

- MySQL `dj_music_radio`；
- RabbitMQ；
- `NeteaseCloudMusicApi:3000`；
- 可选 `DEEPSEEK_API_KEY`。

### 6.5 播放队列持久化

播放队列不是前端临时数组。`MusicPanel.vue` 在播放、加入队列、移除队列、清空队列、切换播放模式时，会调用 `/music/queue/state` 保存：

- `playMode`：`order`、`shuffle`、`repeat`；
- `currentSong`：当前歌曲快照；
- `queue`：队列歌曲快照。

后端写入 `user_play_queue` 和 `user_player_state`。网易云歌曲保存 `source/sourceId/title/artist/album/coverUrl/filePath/duration` 等快照，刷新页面后仍能恢复队列。

## 7. module-rec：每日推荐模块

### 7.1 推荐能力

- `/rec/hot`：读取 Redis ZSET `song:hot:rank`，按分数返回 TOP10；
- `/rec/behavior`：接收用户行为，写 MySQL，并按 action 增加 Redis 热度；
- `/rec/daily`：查询当天 `daily_recommend`；
- `/rec/daily/refresh`：按最新行为手动刷新当天推荐；
- `/rec/similar`：相似歌曲推荐；
- `/rec/preferences`：根据用户行为统计流派偏好。

### 7.2 相似推荐策略

```text
输入 songId
  -> Feign 调 module-music 获取歌曲情绪、流派、歌手、专辑、时长
  -> 优先搜索情绪标签相近歌曲
  -> 再搜索有效流派和同歌手歌曲
  -> 再按 user_behavior 做协同过滤
  -> 最后用同专辑、相近时长补足
```

### 7.3 行为计分

| 行为 | 热度增量 |
|:--|--:|
| `play` | +1 |
| `like` | +3 |
| `share` | +2 |
| `skip` | +0 |

### 7.4 定时任务

`RecommendScheduler` 每天 02:00 执行：

- 找到有播放/收藏行为的用户；
- 取用户 Top 歌曲；
- 按情绪标签、有效流派、歌手、协同过滤、专辑和时长生成最多 10 条每日推荐；
- 删除当天旧推荐，写入新推荐；
- 通过 RabbitMQ 发推荐已更新通知。

## 8. module-user：用户中心

`module-user` 已从最初占位服务整合为阶段版真实用户中心，端口 8084，通过 Gateway 暴露 `/user/**`。

### 8.1 当前能力

- `/user/register`：注册用户，用户名唯一；
- `/user/login`：校验 BCrypt 密码并签发 JWT；
- `/user/info`：根据 `Authorization: Bearer <token>` 查询或更新当前用户资料；
- `/user/password`：校验原密码后修改当前用户密码；
- `/user/favorite/list`、`/user/favorite/{songId}`：在校验用户 token 后调用 music 模块收藏接口；
- `/user/history`、`/user/history/add`：在校验用户 token 后调用 music 模块播放历史接口。

### 8.2 数据边界

用户身份数据保存在：

```text
mall.dj_user
```

收藏和播放历史不在 user 模块重复建表，继续复用 music 模块的数据：

```text
dj_music_radio.user_favorite
dj_music_radio.play_history
```

这样做的原因是 music 模块已经基于收藏、播放历史、歌词情绪和用户品味形成联动，user 模块如果再维护一套收藏/历史表，会导致播放器、推荐和用户中心看到的数据不一致。

### 8.3 仍需说明的限制

当前 JWT 校验只在 `module-user` 自己的接口中完成。Gateway 还没有统一鉴权过滤器，`module-chat`、`module-music`、`module-rec` 仍主要通过前端传入的 `userId` 做业务联动。答辩时可以讲“已具备用户会话和用户中心”，但不要说全站所有接口都已经被 Gateway 强制鉴权。

## 11. 当前风险和后续待办

| 风险/待办 | 影响 |
|:--|:--|
| Gateway 未统一鉴权 | 用户接口已校验 JWT，但 chat/music/rec 尚未通过网关强制鉴权 |
| Java 版本未统一 | 父 POM 是 25，默认 Java 是 17，必须按手册构建 |
| `module-rec` SQL 已补 | 新人需执行 `module-rec/src/main/resources/sql/rec_tables.sql` 初始化推荐表 |
| `/music` 前端直连 8082 | 开发方便，但最终部署要决定是否全部走 Gateway |
| 数据库数据不随 Git 同步 | 新人拉仓库后歌库可能为空，需导入歌单 |
| 网易云播放依赖外部服务和版权状态 | 部分歌曲无 URL 属于正常外部限制 |
| DeepSeek/和风天气需要个人 key | 未配置会显式降级 |
| 用户模块仍无验证码、找回密码、真实头像上传 | 当前只实现注册登录、资料、修改密码和收藏/历史入口 |

## 12. 答辩可讲亮点

- 微服务拆分清晰：chat/music/rec/user/gateway；
- AI 不是直接编歌，而是通过工具调用真实数据；
- WebSocket 支持实时对话和前端工具请求；
- 天气和定位工具可接入对话上下文；
- 音乐模块支持网易云导入、歌词、情绪画像；
- 推荐模块有 Redis 热榜、行为采集、相似推荐和定时任务；
- 前端实现单屏音乐控制台，聊天推荐可直接点击播放。

## 13. 演示路线建议

答辩时建议按下面顺序演示，能把各模块联动讲清楚：

1. 打开前端首页，说明这是单屏 DJ 控制台；
2. 展示顶部时间和天气，悬浮查看详细天气；
3. 在聊天区问普通问题，证明不是只能关键词模板；
4. 问“我这里天气怎么样，适合听什么”，触发浏览器定位和天气工具；
5. 点击聊天推荐歌曲，让中间播放器开始播放；
6. 在音乐面板搜索网易云歌曲，展示播放 URL、歌词和封面；
7. 导入网易云歌单，说明 RabbitMQ 异步导入；
8. 展示歌曲情绪或歌单情绪总览，说明 DeepSeek + 关键词兜底；
9. 播放或收藏后看推荐面板，说明 Redis 热榜和偏好标签；
10. 登录或注册一个用户，展示底部用户状态栏、收藏/历史弹窗和跨模块刷新；
11. 最后说明当前 JWT 已用于用户中心，后续可继续补 Gateway 全局鉴权。
